BORA NA TECH - PLANO DETALHADO DE IMPLEMENTACAO DO BACKEND
===========================================================

Objetivo deste documento
------------------------
Este documento transforma a varredura do frontend em um plano completo de implementacao.

A leitura principal e:

1. O frontend atual NAO representa um site quase estatico.
2. O frontend representa uma plataforma viva, com conta, assinatura, plano Pro, conteudo dinamico, favoritos, progresso, diario de estudos, ferramentas de IA, busca, admin, integracoes e possivelmente automacoes.
3. O backend atual e minimo: Express, uma rota de IA e servidor de arquivos estaticos.
4. A maior parte dos dados hardcoded/mockados no frontend deve ser migrada para banco, CMS, APIs internas ou integracoes externas.
5. O plano abaixo considera o caminho mais coerente com o que ja existe: manter Supabase Auth como identidade, usar Postgres/Supabase para dados do produto e ampliar o Express como backend seguro para operacoes sensiveis, webhooks, IA e integracoes.

Principio arquitetural
----------------------
O backend deve ser dividido em quatro camadas:

1. Identidade e autorizacao
   Supabase Auth pode continuar sendo a fonte de identidade.
   O backend Express deve validar JWT do Supabase nas rotas privadas.

2. Dados do produto
   Postgres/Supabase deve guardar perfis, favoritos, progresso, diario, assinaturas, conteudo e historicos.

3. Operacoes sensiveis
   Express deve cuidar de billing, webhooks, IA, rate limit, integracoes externas, jobs e admin.

4. Conteudo vivo
   Conteudos que hoje estao em TS devem virar registros editaveis ou sincronizados.
   Podem comecar no Postgres e depois evoluir para CMS se necessario.


Decisao recomendada de stack
----------------------------
Como o projeto ja usa Supabase Auth, a abordagem mais direta e:

- Supabase Auth para login/cadastro/reset/sessao.
- Supabase Postgres para dados.
- Supabase RLS para proteger dados acessados diretamente pelo client, quando fizer sentido.
- Express para:
  - IA.
  - Billing.
  - Webhooks.
  - Admin sensivel.
  - Integracoes externas.
  - Jobs/cron.
  - Endpoints agregadores.
  - Validacao Pro server-side.

Essa arquitetura evita reescrever a autenticacao e aproveita o que ja existe, mas corrige o problema central: hoje o cliente sabe quem esta logado, mas o servidor nao usa essa informacao para proteger recursos.

Autenticacao no backend
-----------------------
Problema atual:
- O frontend sabe se existe user/session via Supabase.
- O backend /api/ai/:tool nao exige usuario.
- Logo, qualquer pessoa pode chamar ferramentas de IA se souber a rota.

Implementacao necessaria:

1. Criar middleware auth opcional/obrigatorio.

Fluxo:
- Cliente envia Authorization: Bearer <access_token_do_supabase>.
- Backend valida token.
- Backend injeta req.user com:
  - id
  - email
  - role
  - raw token claims

2. Criar helper requireAuth.

Responsabilidade:
- Bloquear rota privada se token nao existir ou for invalido.
- Retornar 401.

3. Criar helper requirePro.

Responsabilidade:
- Depois de autenticar, consultar assinatura ativa.
- Permitir se status ativo/trialing e current_period_end valido.
- Retornar 403 se usuario nao for Pro.

4. Criar helper requireAdmin.

Responsabilidade:
- Validar role admin no banco ou claim custom.
- Nunca usar localStorage como sessao admin real.

RLS e seguranca no Supabase
===========================

Regras gerais:
- profiles: usuario so le/edita o proprio profile.
- subscriptions: usuario le a propria assinatura; escrita apenas service role/webhook.
- user_bookmarks: usuario CRUD apenas nos proprios favoritos.
- study_entries: usuario CRUD apenas nas proprias entradas.
- ai_usage_logs: usuario pode ler talvez agregados proprios; escrita apenas backend.
- ai_generations: usuario le/deleta apenas as proprias; escrita backend.
- admin_roles: leitura/escrita restrita.
- conteudo publicado: leitura publica.
- conteudo admin/draft: apenas admin.

Exemplos conceituais:
- using (auth.uid() = user_id)
- with check (auth.uid() = user_id)

Operacoes com service role:
- Webhooks de billing.
- Sync externo.
- Admin server-side.
- Escrita de ai logs.

  - apenas IA Pro?
  - limite free mensal?
  - detalhe de area Pro?
  - diario Pro?

4. Retencao de dados de IA
- Curriculo e vaga podem conter dados sensiveis.
- Decidir se salva historico.
- Se salvar, permitir apagar.

5. Conteudo local como fallback
- Bom durante transicao.
- Ruim se virar fonte paralela permanente.
- Definir data.ts como seed, nao como fonte final.

6. RLS direto no client vs API Express
- Dados simples podem ir direto via Supabase com RLS.
- Operacoes sensiveis devem passar pelo backend.

7. Admin
- Pode ser simples no inicio.
- Nao deve usar localStorage como permissao.


Checklist final de "site funcional"
===================================

Conta:
- Usuario cadastra.
- Usuario loga.
- Usuario recupera senha.
- Usuario tem profile persistente.

Plano:
- Usuario ve plano atual.
- Usuario assina Pro.
- Webhook confirma assinatura.
- Usuario Pro acessa funcionalidades Pro.
- Usuario free recebe bloqueio real.

IA:
- IA exige auth.
- IA exige Pro quando aplicavel.
- IA tem limite.
- IA registra uso.
- IA nao expoe chave no browser.

Favoritos:
- Favoritos persistem no banco.
- Favoritos aparecem no perfil.
- Favoritos sincronizam entre dispositivos.

Diario:
- Entrada salva.
- Entrada apaga.
- Streak e metricas persistem.

Conteudo:
- Areas/cursos/projetos/roadmaps/etc vem de API ou banco.
- Admin consegue editar.
- Conteudo publicado aparece.
- Conteudo despublicado some.

Busca:
- Busca consulta backend.
- Resultados vem de conteudo vivo.

Admin:
- Admin usa usuario real.
- Admin tem permissao real.
- Admin gera auditoria.

Integracoes:
- Noticias/eventos/vagas podem ser sincronizados.
- Status da sincronizacao aparece.

Seguranca:
- Rotas privadas exigem token.
- Rotas Pro exigem assinatura.
- Admin exige role.
- Service role nunca vai ao cliente.
- Webhook valida assinatura.
- Rate limit protege IA.


Conclusao
=========
O BORA NA TECH deve ser tratado como uma plataforma dinamica.

O backend atual e apenas uma base minima para IA e hosting do frontend. Para o produto ficar funcional, o centro do trabalho e criar uma camada real de produto:

- identidade reconhecida no servidor
- perfil persistente
- assinatura real
- autorizacao Pro
- dados do usuario
- conteudo vivo
- admin
- busca
- integracoes
- IA controlada

O caminho mais seguro e incremental e:

1. Fazer servidor reconhecer usuario.
2. Criar profiles.
3. Criar assinatura e Pro real.
4. Proteger IA.
5. Persistir favoritos e diario.
6. Migrar conteudo mockado para banco.
7. Criar admin.
8. Adicionar busca e integracoes.

Depois disso, o frontend deixa de ser uma demonstracao rica e passa a ser um produto real sustentado por backend.
